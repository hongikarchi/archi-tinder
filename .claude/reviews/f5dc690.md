## Security Analysis

In-depth walk:

- **Claim mutation hardening (cycle 1 Fix 1, load-bearing)**: `OfficeClaimSerializer` only accepts `proof_text` (max 2000 chars). Previously `contact_email` + `website` were in schema and the view wrote them to live Office row PRE-verification. Reviewer + security flagged this as abuse vector: any authenticated user could overwrite contact fields on any unclaimed Office, AND values stayed after admin rejection. Cycle 1 fix removes both fields from schema entirely. View at `views.py:87-88` uses `update_fields=['claim_status', 'updated_at']` exclusively — surgical update, no other field touched. Regression test `test_claim_does_not_mutate_office_fields` at `tests/test_phase13_office.py:306-334` submits attacker payload `{contact_email='attacker@evil.com', website='https://evil.com'}` and asserts Office fields unchanged + 200 response + claim_status='pending'. Excellent.

- **Throttle on claim** (cycle 1 Fix 4): `OfficeClaimThrottle(UserRateThrottle, scope='office_claim', rate='5/hour')` at `throttles.py`. Mirrors `DevLoginThrottle` inline-rate pattern from `apps/accounts/views.py`. Prevents admin-queue flood from a single authenticated abuser. `cache.clear()` in test isolation prevents cross-test counter leakage.

- **Permissions correctly scoped**:
  - `OfficeDetailView`: `AllowAny` — public read, design intent (browse-without-account; Phase 14 reading flow).
  - `OfficeClaimView`: `IsAuthenticated` — claim requires Django User context for admin out-of-band correspondence post-cycle-1.
  - `OfficeAdminQueueView`: `IsAdminUser` — admin queue gated.
  - `OfficeAdminVerifyView`: `IsAdminUser` — admin verify gated. Validates `claim_status` against allowlist `('verified', 'rejected')` BEFORE write at `views.py:111-115` — defense-in-depth (model already has CHOICES validator, but view-layer allowlist prevents arbitrary value writes via PATCH body).

- **Raw SQL safety** at `views.py:32-39`: parameterized via psycopg2 `[building_ids]` list passed to `cur.execute(query, [building_ids])`. No string interpolation. `building_id = ANY(%s)` is a standard pgvector/PostgreSQL pattern. Safe from injection.

- **`row_map` ordering preservation** at `views.py:42`: SQL `ANY()` returns rows in non-deterministic order; the dict-then-iterate pattern preserves the `OfficeProjectLink.confidence DESC` ordering from the prior ORM query at lines 21-26. Correctness-preserving.

- **No new external dependency** — uses Django ORM + psycopg2 stdlib + DRF (already in codebase). UUID PK uses `uuid.uuid4` (stdlib). No new pip packages.

- **No PII leakage in cache key** — there's no caching layer in PROF1 endpoints (rate limiter caches throttle counters internally; user data not cached).

- **Logging safety** — no `logger.info` / `logger.warning` calls in views.py — claim path emits no log line (might be a sub-MINOR observability gap, but not a security issue).

- **Migration 0002 confidence bounds**: `MinValueValidator(0.0)` + `MaxValueValidator(1.0)` at `models.py:69-71` are Python-layer; not DB constraint. Validates on `full_clean()` only. Test `TestOfficeProjectLink` covers bounds via `full_clean()`. ORM `.create()` bypasses validation by default; admin queue / future ETL would respect validators when going through serializer/Form path. Acceptable — DB constraint would be ideal for defense-in-depth but Python-layer is the standard Django pattern.

- **claim_status enum allowlist** at views `views.py:111-115`: allowlist intersection with model CHOICES — defense-in-depth. Even if model CHOICES expands, view stays restricted to `('verified', 'rejected')` until explicit code change.
