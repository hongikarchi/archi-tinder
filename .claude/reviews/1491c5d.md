## Security Analysis

In-depth walk of the auth / token / input-validation / authorization surface touched by this branch:

- **No new auth/authz surface**. IMP-8 flag is server-side only (Django settings), not exposed via API.
- **No new request-body shape**. SwipeView.post payload is unchanged. Response payload is unchanged in shape; only `prefetch_image` / `prefetch_image_2` values are nullable now (already nullable in pre-IMP-8 codepaths — session end, dislike-fallback path).
- **Cache key collision**: `prefetch:{session_id}:{cache_round}` (UUID + int) vs DRF throttling's `throttle_<scope>_<ident>`. Different namespaces, no collision (verified in commit body + tested in `TestBackwardCompat::test_flag_off_no_cache_write` which clears the cache and asserts no `prefetch:*` keys exist).
- **Cache contents**: building_ids (already public in API responses) + ISO timestamp. No PII. No tokens, no secrets.
- **No raw SQL changes**. Engine module SQL surface unchanged. No new injection vectors.
- **Error logging via `logger.warning`** does not include any secret material — only the exception message + cache key.
- **Daemon thread**: `daemon=True` ensures bg thread does not survive process shutdown. No long-running zombie processes; no resource leak.
- **Connection lifecycle**: `connection.close()` at start releases parent's thread-local connection (which may already be closed by the time bg thread runs); `finally` block releases bg thread's own connection. Test `TestConnectionClosureOnExit::test_connection_close_called_in_finally_on_exception` verifies the finally runs even on exception. No connection pool exhaustion risk.
- **Multi-worker prod**: with LocMemCache, the bg thread in worker A writes to cache; next swipe arriving on worker B would cache-miss. Not a security issue (just a correctness/efficiency one) — primary path would run standalone, same as today. Documented in settings comment + commit body. Mitigation: settings-only swap to django-redis at production deploy time.
- **Settings-flag pattern**: `async_prefetch_enabled = False` default. Existing 260 baseline tests verified flag OFF is byte-identical to pre-IMP-8 (`TestBackwardCompat`).

## Observations (sub-MINOR, not flagged as findings)

1. **`saved_pool_embeddings = pool_embeddings`** at views.py:889 is a reference, not a copy. The dict itself is constructed fresh per-request by `engine.get_pool_embeddings(saved_pool_ids)` (returns a new dict containing references to numpy arrays from IMP-7's per-building-id cache). Numpy arrays are immutable in our usage. Not a race risk, but the asymmetry (`list(...)` for the other snapshots, plain `=` for this one) is worth a code comment in a future cleanup pass — not blocking.

2. **Half A user-perceived UX**: with `async_prefetch_enabled=True`, response is 310ms faster but `prefetch_image=None` means the frontend cannot pre-fetch the next card's image URL via `<img>` preload. The next swipe will start the image fetch from cold. Trade-off is real but documented; Half B addresses it. This is consistent with the spec framing ("310ms savings per spec come from REMOVING prefetch FROM the response path") — semantically aligned.

3. **Cache key uniqueness across concurrent rounds**: with the async path, the bg thread for round N might still be running when the swipe for round N+1 arrives. The two would write different keys (`prefetch:{uuid}:{N+1}` vs `prefetch:{uuid}:{N+2}`) so no collision; same-key races would only happen if the same round were processed twice (impossible — `current_round` increments inside the transaction). Correctness-safe.

4. **Possible `TestConnectionClosureOnExit` skipped class**: the file has 6 classes but pytest reports `1 skipped` — likely `TestConnectionClosureOnExit` whole class was missing `@pytest.mark.django_db` and got skipped, or one specific test inside is intentionally skipped. Worth a quick verify in a follow-up but doesn't block this review (the class's purpose is verified separately via direct-call mocks; skip is benign).

These are deliberately not raised as MINOR findings — they're notes for future awareness, not gaps that should block this push.
