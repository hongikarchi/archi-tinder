## Security Analysis

In-depth walk of the auth / token / input-validation / authorization surface touched by this branch:

- **No new auth/authz surface**. `stage_decouple_enabled` flag is server-side only.
- **No new request-body shape**. parse_query / parse_query_stage1 signatures unchanged from a caller perspective.
- **Response shape preserved**. Stage 1 returns same dict shape with `visual_description=None`; SessionCreateView reads V_initial from cache when present, falls through gracefully.
- **V_initial cache key safety** at `services.py:117-127`: SHA-256 hash of `raw_query` (16 hex prefix) before key inclusion. Raw text never stored in cache key/value/logs. user_id from authenticated `request.user.id` — no IDOR vector (cross-user cache pollution prevented by user_id in key).
- **Input validation at boundary**: `_raw_query_early` coerced to None for non-string + 1000 char hard cap (`views.py:268-273`). Same defensive pattern as Topic 01 RRF. Defense-in-depth even though Stage 2 already filters by user_id.
- **Cache contents**: 384-float V_initial vector. No PII, no tokens, no secrets.
- **HF API call at Stage 2**: existing path (already in codebase from Topic 03 HyDE). `HF_TOKEN` is env-var loaded; commit body confirms NO new external dependency. urllib.request is stdlib.
- **HF HTTPError response body capture (first 200 bytes)** in failure SessionEvent's `error_message` field — flagged as security MINOR in 7348593 commit body. HF inference-API errors do NOT echo Authorization header so no token leak in practice; tightening to log only `http_status` + fixed-string reason is documented as future hardening. Acceptable as-is.
- **Stage 2 thread isolation**: daemon=True (no zombie processes); finally `connection.close()` (no DB connection leak — each thread gets its own connection); all exceptions caught (no error bubbles to user).
- **Migration 0014**: AlterField on `event_type` choices only. No PostgreSQL schema change. Reverse-safe.
- **Cache key collision safety**: `'v_initial:{user_id}:{sha256[:16]}'` (user_id integer + 64-bit SHA-256 prefix) provides 64-bit collision resistance ≫ session count. Safe for production scale.
- **No raw SQL changes**. Engine module SQL surface unchanged.
- **80% TTL invariant** (IMP-5, c133787) untouched. V_initial cache TTL is independent (1h, separate key namespace).
- **DRF throttling cache compatibility**: V_initial uses `'v_initial:'` prefix; DRF throttling uses `'throttle_<scope>_<ident>'`; IMP-5 uses `'gemini_cache_name:'`; IMP-8 uses `'prefetch:'`. Distinct namespaces — no collision.
- **`stage_decouple_enabled` default OFF** — production behavior unchanged until empirical staging validates the flag flip.

### Why the v1.9 boundary still produces sys artifacts on multi-turn

When Gemini delivers card+session-creation in the same response that fires the clarification bubble (observed pattern this cycle and last), the `waitCardOrClarification` polling returns 'clarification' on iteration N and 'card' on iteration N+1 milliseconds later. `t_last_user_submit` was set just before the canned-reply Enter press; the card was visible within ~3 ms of polling resumption. Mathematically correct per v1.9 boundary; practically meaningless because the card was already there.

This is a known harness artifact (documented in 1b2bd21.md Part B and 1b2bd21-improvements.md Tier 4). Not a code regression in 7348593.

Per commit body, `stage_decouple_enabled=False` is the default. This Part B run measured **pre-IMP-6 baseline behavior**. The structural latency fix (Stage 1 schema enforcement reducing output tokens ~290-400 → ~150-220) is dormant — only activates when the flag is flipped in staging/production.
