- All 3 runs: `card_not_visible_18s` (ttfcs list empty → mechanical fail).
- Root cause via `failhtml` inspection: the Korean query triggered a **multi-turn AI clarification dialog** at `/search`. AI replied: "친환경 주거 건축의 방향성을 좁혀볼게요: 자연 재료를 적극적으로 사용한 유기적 형태가 끌리세요, 아니면 첨단 기술과 시스템을 강조한 절제된 형태가 끌리세요?" — i.e., parse-query returned a clarifying question (`Last call: POST /parse-query/ 200 2767ms`, debug overlay shows `Session: none`), and no session was ever created because the harness only sent one turn.
- This is a **harness gap**, not a regression. The harness was authored against the prior single-turn flow.

**BareQuery** (`modern`, gate 5000 ms):
- All 3 runs: `card_not_visible_18s` — same root cause as Korean. Bare 1-word queries are even more likely to trigger clarification.
- API call breakdown: `parse-query/` 3040 / 3453 / 6392 ms, all 200 — successful clarification responses, just not session-creating.

### Failure-cause attribution vs. commit 0394220

Commit 0394220 modifies **only** these files: `engine.py`, `event_log.py`, `models.py`, `views.py`, `migrations/0013_*`, `tests/test_imp10_topic06_telemetry.py`. None of these touch:
- The `/parse-query/` endpoint or its multi-turn UX
- The `/search` frontend page or AI dialog flow
- Any latency-affecting hot path (no DB query shape changes, no Gemini hyperparameters, no embedding logic)

The 0394220 diff added (a) the `_clustering_agg` aggregation call inside `session_end`, (b) the `engine.get_last_clustering_stats()` read inside `confidence_update`, (c) `_cosine_top10/_gemini_top10/_dpp_top10` capture inside `SessionResultView`, and (d) the `compute_corpus_rank` SQL inside `ProjectBookmarkView`. All four are off the swipe hot path entirely; the first three are session-end-only telemetry, the fourth runs only on bookmark POST. **None of them affect time-to-first-card.**

Conclusion: Part B FAIL is the conjunction of two pre-existing conditions, both unrelated to this branch:
1. **Multi-turn AI clarification UX** (introduced earlier on origin/main) — harness needs an update to detect the AI follow-up bubble and answer it (or use queries known to skip clarification, like Brutalist).
2. **Structural latency floor** for queries that DO bypass clarification (Brutalist) — `parse-query/` Gemini RTT (~2.7-3.0 s) + `sessions/` initial pgvector + DPP pipeline (~1.3-1.9 s) hits the 4000 ms ceiling. This is the same Neon-RTT + Gemini-cold-call structural floor flagged in prior cycles `f607e73`, `57b3244`, `2da9c65`, `e391c95` — IMP-8 / INFRA-1 (regional Neon move + thread-pool prefetch) are the mitigation candidates and have not yet shipped.

### Recommended follow-up (for main implementation terminal)

Captured in detail at `.claude/reviews/0394220-improvements.md`:
- **Tier 1 — Harness update** — teach Step B3-B4 to detect the AI clarification bubble and either (a) answer it with a fixed canned reply, or (b) use only queries known to skip clarification, or (c) measure the TTFC across the entire dialog inclusive of clarification turns. Without this, every Korean / bare query review will mechanically FAIL on B4 even when the diff is fine.
- **Tier 1 — Re-evaluate spec §4 TTFC budget** — Brutalist hits 4314-6846 ms repeatedly across cycles. Either (a) the 4000 ms budget is unrealistic given current Neon RTT + Gemini cold-call without IMP-8/INFRA-1, and the spec should reflect that until those ship, or (b) IMP-8 (async-thread prefetch of next-card embeddings) needs to be prioritized.
- **Tier 2 — Multi-turn UX measurement** — if the multi-turn flow is the intended end-state, the spec §4 budget should be redefined as "first-card-visible from the user's *last* free-text submit before the session is created" rather than "first submit", since clarification turns are user-driven, not system-driven.
