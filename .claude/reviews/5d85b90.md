## Findings (Part A)

### 1. [MINOR] `orchestrator.md` Step 2.5 hardcodes `recommendation` app in showmigrations check
- **File:** `.claude/agents/orchestrator.md` Step 2.5 (the `manage.py showmigrations` invocation, around line 100)
- **Axis:** 2 (Correctness & logic depth) / 5 (Code quality — internal inconsistency)
- **Issue:** Step 2.5 fires when ANY file under `backend/apps/*/migrations/` is touched (per its own trigger condition: "If ANY file under `backend/apps/*/migrations/` was created or modified"). But the verification command is:
  ```bash
  cd backend && python3 manage.py showmigrations recommendation 2>&1 | grep -E '\[ \]'
  ```
  Filtering by `recommendation` means migrations in `accounts/`, or any future app, are unchecked. The trigger condition is broad ("any app"), but the check is narrow (one app). Compare with `review.md` Step B1bb, which is consistent with itself:
  ```bash
  cd backend && python3 manage.py showmigrations 2>&1 | grep -E '\[ \]'
  ```
  (no app filter — checks all apps).
- **Why it matters:** For the current scope, only `recommendation` has migrations being changed, so the bug is latent. But once an `accounts/` or future-app migration is created without applying, Step 2.5 silently passes and the gap reopens — the postmortem would then need to be re-relearned. The trigger and the check should agree.
- **Suggested fix:** Drop the `recommendation` argument:
  ```bash
  cd backend && python3 manage.py showmigrations 2>&1 | grep -E '\[ \]'
  ```
  One-character/word change in `.claude/agents/orchestrator.md` Step 2.5. Brings it in line with `review.md` Step B1bb.

Either reading: **FAIL**.

### Secondary signal (correctness, not gate-level)

Same backend code, same request body, very different responses.
