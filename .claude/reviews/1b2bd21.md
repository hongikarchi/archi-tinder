### Why the v1.9 sys metric is an artifact for multi-turn flows

The harness's `waitCardOrClarification` polls the page on each loop iteration. When Gemini delivers card+session-creation in the same response that fires the clarification bubble (a Gemini behavior observed in this cycle's data), the wait loop sometimes returns 'clarification' on iteration N (bubble appeared first) and then immediately returns 'card' on iteration N+1 (~few ms later, since the card was already rendered when iteration N+1's polling started).

When this happens, `t_last_user_submit` was set just before the canned-reply Enter press, and the card was visible within milliseconds — producing sys=2-3 ms which is mathematically correct per the v1.9 boundary (`t_first_card_visible - t_last_user_submit`) but practically meaningless because the card was already there.

This is a harness-measurement artifact specific to Gemini's "clarification + speculative session-creation" delivery pattern. Not a code regression.

### Failure-cause attribution vs. diff

- **Brutalist PASS** is the intended empirical signal that M1 mitigation works: clarification rate dropped from 3/3 to 0/3 on this query class.
- **Korean + BareQuery sys_p50 artifacts** are harness measurement issues, not code regressions:
  - 1 of 3 runs hit the harness's MAX_TURNS=3 cap (Korean run 1, BareQuery run 3) and returned undefined
  - The other 2 runs measured sys=2-3 ms because Gemini delivers card+clarification in the same response
  - Real total_user_felt latency is 5500-8000 ms (gate would FAIL even if all 3 runs returned numeric values)
  - This confirms the same structural Gemini parse-query floor (~2900-3500 ms per call × 2-3 turns = 6-10 s total) flagged in prior cycles
- **The diff itself is clean** — Static review verdict 0/0/0 carries the verdict.
