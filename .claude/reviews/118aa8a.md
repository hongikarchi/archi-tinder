### 1. [MAJOR] Slash command and subagent duplicate ~90% of their content
- **File:** `.claude/commands/deep-review.md` (175 lines), `.claude/agents/deep-reviewer.md` (175 lines)

- **Issue:** Frontmatter, opening paragraph, and the invocation-contract framing differ — but Steps 1–5, the 7-axis checklist text, the report-format template, and the Rules block are byte-identical or near-identical. Substantive overlap ≈ 90%.
- **Why it matters:** Any checklist change (new axis, reworded criterion, updated severity rules) requires editing both files in sync. Single-file edits silently drift between the two entry points: slash-command callers and programmatic subagent callers get different behaviors. This will happen within a few iterations.
- **Suggested fix:** Pick one of three approaches:
  - (a) **Single source of truth** — make `.claude/agents/deep-reviewer.md` canonical; replace the body of `.claude/commands/deep-review.md` with a short directive to spawn the subagent with the supplied range. Loses in-session execution but removes duplication.
  - (b) **Shared include** — extract Steps 1–5 + checklist + report format + rules into `.claude/shared/deep-review-body.md`; each entry file opens with "READ .claude/shared/deep-review-body.md then follow those instructions."
  - (c) **Accept + lint** — add a `# DO NOT EDIT DIRECTLY — keep in sync with {other file}` header and a pre-commit hook that diffs the shared regions.

### 3. [MINOR] `.gitignore` does not defensively cover future `.claude/` subpaths
- **File:** `.gitignore:4-9`
- **Axis:** 4 (Security in depth) — forward-looking
- **Issue:** The targeted exclusion list (`settings.local.json`, `plans/`, `reviews/`, `projects/`, `cache/`) covers what Claude Code currently writes. - **Why it matters:** The former wholesale `.claude/` exclusion was a blunt but safe default; the targeted replacement is more permissive by design. Forward-compatibility is now our responsibility.
- **Suggested fix:** Prefer a whitelist pattern over an enumerated blacklist. Replace lines 4–9 with:
  ```

  !.claude/Goal.md
  !.claude/Task.md
  !.claude/Report.md

  !.claude/agents/**
  !.claude/commands/
  !.claude/commands/**
  ```
  Anything we haven't explicitly opted into stays ignored, closing the future-unknown hole.

### 5. [MINOR] Hardcoded model identity in report template
- **File:** `.claude/commands/deep-review.md:111`, `.claude/agents/deep-reviewer.md:112`
- **Axis:** 5 (Code quality)
- **Issue:** The template literally says `Reviewer:** Claude (Opus 4.7, /deep-review)`. The slash-command frontmatter does not pin a model (no `model:` field), so whatever session model is active will run. On a future session using Sonnet or a newer Opus, the report still claims Opus 4.7.

### 6. [MINOR] No handling for missing `origin/main` ref or invalid user-supplied range
- **File:** `.claude/commands/deep-review.md:27-43`, `.claude/agents/deep-reviewer.md:26-44`
- **Axis:** 2 (Correctness & logic depth)
- **Issue:** If the repo has no `origin` remote, or `main` is not tracked on that remote, `git log origin/main..HEAD` fails with "unknown revision: origin/main". A user typo like `/deep-review bogus..HEAD` hits the same error class. The workflow doesn't catch either case — it fails mid-Step-1 with a raw git error.
- **Why it matters:** Poor first-run UX on fresh clones or CI-like environments; confusing failure on typos. The workflow's nice abort message (`DEEP REVIEW: nothing to review — HEAD matches origin/main`) only fires when the range is *valid but empty*.
- **Suggested fix:** Add a precheck before the main command set: `git rev-parse --verify "$RANGE_LHS" 2>/dev/null` where `RANGE_LHS` is the left side of the range (default `origin/main`). On failure, abort with `DEEP REVIEW: could not resolve range '<range>'. Ensure the remote is configured ('git fetch origin main') or pass an explicit local range (e.g. /deep-review HEAD~5..HEAD).`

### 7. [MINOR] `git diff` two-dot vs three-dot inconsistency
- **File:** `.claude/commands/deep-review.md:38,47`, `.claude/agents/deep-reviewer.md:37,48`
- **Axis:** 2 (Correctness & logic depth)
- **Issue:** The default range `origin/main..HEAD` (two-dot) is applied uniformly to both `git log` and `git diff` via the `<range>` placeholder. For `git log`, two-dot is exactly right (commits reachable from HEAD but not origin/main = unpushed work). For `git diff`, two-dot shows "changes to turn origin/main into HEAD as snapshots" — which includes any commits that were *on* origin/main but not on HEAD as *removals* in the diff. In linear main-direct flow (current state), two-dot and three-dot coincide; in any divergent flow (after a rebase-before-push, for example), two-dot inflates the review scope incorrectly.
- **Why it matters:** Silent correctness footgun once the repo adopts any branch/rebase workflow.
- **Suggested fix:** Separate the log range and the diff range in Step 1: use `origin/main..HEAD` for `git log` and `origin/main...HEAD` (three-dot) for `git diff --stat` and `git diff --name-only` in Step 2. Document the distinction in the prose.

- **Role orthogonality** — `reviewer` (API/logic/perf PASS/FAIL) and `security-manager` (SQLi/XSS/auth keyword scan) remain the orchestrator's commit-gate inside the fix loop. `deep-reviewer` is the **push-gate**, invoked manually on a separate terminal, never entering the fix loop.
