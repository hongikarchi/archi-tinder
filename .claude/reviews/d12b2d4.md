### 2. [MINOR] Push-failure-then-rebase edge case leaves a stale `REVIEW-PASSED` signal in Handoffs
- **File:** `.claude/agents/deep-reviewer.md:253-257`, `.claude/agents/orchestrator.md:110-131`, `CLAUDE.md` Code Review section
- **Axis:** 2 (Correctness & logic depth) / 1 (Architecture alignment)
- **Issue:** The protocol is explicit that after `REVIEW-PASSED`, push is user-initiated and push failures are user-resolved ("no follow-up signal from you, no retry by you" — `deep-reviewer.md:257`). But if push fails non-ff (because remote drifted in the narrow window between Step 6c and user's `git push`), the correct recovery is `git pull --rebase` + `git push`, and rebasing **rewrites local commit SHAs**. The `REVIEW-PASSED: <old_sha>` entry stays in Task.md Handoffs, pointing at a SHA that no longer exists locally, while the newly-rewritten commits pushed to `origin/main` have never been reviewed at their rewritten SHA. The orchestrator (`orchestrator.md:260-262`) treats `REVIEW-PASSED` as cycle-closing and starts the next task.
- **Why it matters:** Narrow-window edge case (drift happens in the seconds between 6c and the user's manual `git push`), but when it fires the protocol silently lets unreviewed rewritten code reach `origin/main`. Relies entirely on user discipline to re-run `/deep-review` after a push failure + rebase.
- **Suggested fix:** Add one line to `deep-reviewer.md`'s Step 6d (and mirror to the slash command) after the existing warning about push failures: `If push fails non-ff and you rebase (rewriting local SHAs), the REVIEW-PASSED signal above is stale — re-run /deep-review before retrying git push.` Alternatively, document this in CLAUDE.md's Code Review section as a user-discipline note. Lowest-friction option — pure documentation, no protocol change.
