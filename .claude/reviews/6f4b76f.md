### What needs to be addressed (the real findings)

The swipe-loop FAIL is a NEW class of issue — not workflow budget but actual structural latency. Three plausible interpretations:

1. **Spec §4 swipe-budget is unrealistic for the current Neon-based deployment** — Neon Postgres has ~30-60 ms RTT minimum from this dev machine. With 4-5 sequential RTTs per swipe (lock, embed, swipe-event-create, project-save, session-save), backend total_ms can't go below ~500 ms even with optimal in-process work. Plus prefetch's 2x card-selection adds ~300 ms. The 700 ms ceiling assumes everything is local. **Decision**: loosen swipe budget to 1500 ms (matches what an actual user with this DB would feel) or move to a closer/cached DB.

2. **Swipe-loop has too many sequential DB calls** — could be batched (e.g., merge swipe-event create + project save + session save into one transaction with fewer round trips, or make prefetch async/eager via SSE/WebSocket). Real engineering work, ~1 Sprint.

3. **`get_pool_embeddings` is the hidden cost** — 150 embeddings × 384 floats = ~58KB JSON column fetch over Neon RTT. If pool is already cached server-side, this is fast; on cache miss (first swipe of session, after pool exhaustion escalation, etc.), it's ~200ms+ per fetch. Could be optimized with a session-level memoization scheme.

These options track with `.claude/reviews/57b3244-improvements.md` Tier 2.2 ("tier-aware Part B gates" — separate parse-query budget from swipe budget). Now that Tier 1.2 has landed and Step B4 PASSes, the remaining workflow-vs-reality gap is at Step B5. The decision pattern from prior cycles applies: either loosen the spec budget to match measured reality, or invest engineering work to bring reality under the budget.
