## Optimization Opportunities

- **`_liked_id_only` is O(N) per like swipe** (views.py:25-35). At session-typical scale (≤25 likes per spec §4) this is negligible (sub-ms). For pathologically long sessions (200+ likes), it becomes O(N²) over the session. Out of scope; flag if scaling pattern changes.
- **Migration 0007 `backfill_liked_intensity` does a full-table scan** on `Project.objects.all()`. At current scale fine; if `Project` ever exceeds 100K rows, switch to a paginated `iterator()` + bulk update. Out of scope.
- **Pool score normalization adds 1 division per row** in the SQL — sub-millisecond impact on a ~3,465-row corpus. No action needed.

## Security Analysis

- **`intensity` request input validation is at the right boundary** (views.py:540-550): try/except on `float()` catches TypeError/ValueError; clamp `max(0.0, min(2.0, intensity))` bounds the value; default 1.0 on missing or null. No SQL injection (intensity is numeric, stored in JSONField). No XSS surface (not echoed in HTML).
- **`saved_ids` is read-only via `ProjectSerializer.read_only_fields`** (serializers.py:18-20) — clients cannot directly write this via PATCH. Bookmark endpoint will be a dedicated route with its own authorization checks (not in this batch).
- **Migration runs as a `RunPython` operation** — no SQL injection surface (Django ORM); idempotent guard for re-runs (`isinstance(entry, dict) and 'id' in entry`); silently drops malformed entries with documented rationale.
- **Governance hardening adds defensive layers** — every main/review-terminal subagent's definition now repeats the `research/` read-only rule. A subagent that reads only its own file (not CLAUDE.md) still gets the rule. Defense in depth.
- **`git-manager.md` exclusion patterns verified** to actually exclude nested `research/spec/`, `research/search/`, etc. — empirical test: `git ls-files -- ':(exclude)research/*'` returns 0 research-prefixed paths even though research/ has 13 tracked files including nested ones.
- **No new endpoints**, no new auth surfaces, no JWT/token lifecycle changes.

No vulnerabilities introduced.
