## Security Analysis

In-depth walk:

Algorithm reads `Project.disliked_ids` directly via Django ORM â€” no API leak surface.
- **Visibility gate semantics** at `views.py:213-220`:
  - Owner: 200 with full data (public + private)
  - Anon / non-owner + public: 200 with full data
  - Non-owner + private: 403 (acknowledges existence but blocks contents)
  - Trade-off documented in commit body: 403 reveals existence (vs 404 would not). Acceptable per spec Â§2.
- **PATCH/DELETE owner-only enforcement** at `views.py:223-232`:
  - 401 if no `_get_profile(request)` (no UserProfile)
  - 403 if `project.user_id != profile.pk`
  - Correct ladder; IDOR-safe (cannot mutate other users' projects).
- **`ProjectSelfUpdateSerializer.fields = ['name', 'visibility']`**: explicit allowlist. PATCH cannot mutate `liked_ids`, `saved_ids`, `disliked_ids`, `filters`, `reaction_count`, etc. Reaction_count is owner-write-protected even though the field will eventually have its own write path (Phase 15).

- **`UserProjectsListView` visibility filter** at `views.py:264-265`: `qs = qs.filter(user=target_profile).order_by(...); if not is_owner: qs = qs.filter(visibility='public')`. Non-owner truly sees only public projects.
- **No raw SQL changes**. Engine surface unchanged. `engine.get_buildings_by_ids` is Django ORM helper.
- **Migration 0015 reverse-safe**: pure AddField with defaults. Rolling back drops `visibility`, `reaction_count`, `raw_query` columns cleanly without data loss for the additional columns (existing columns untouched).
