## Security Analysis

In-depth walk of the auth / token / input-validation / authorization surface touched by this branch:

- **No new auth/authz surface**. Both flag and env var are server-side only.
- **HF URL change**: same `HF_TOKEN` Bearer auth (env-var loaded). Same payload format. Same response parsing. No new external dependency. URL string is server-internal constant.
- **Env var read at settings import**: `os.getenv('STAGE_DECOUPLE_ENABLED', 'false')` is read once at process start. No user input, no injection vector. Default 'false' is safe (production behavior preserved).
- **Strict env var parsing**: only `'true'` (case-insensitive) flips True; anything else (typos, unexpected values) reads False. Defensive default — if Railway misconfigures the env var, behavior is byte-identical to default OFF.
- **validate_imp6 management command**: requires shell access to invoke. In-process flag override with try/finally restore + cache.clear() at end. Reads SessionEvents via ORM (no raw SQL). Output written to gitignored `_validation_imp6.md`. No production side effects.
- **HF API call security**: existing path (now with corrected URL). Authorization header carries `HF_TOKEN`; verified in commit body that HF inference API responses do NOT echo the Authorization header in error bodies (same as Sprint C 7348593 review's documented MINOR — pre-existing observation).
- **No raw SQL changes**. Engine module SQL surface unchanged.
- **No new database fields**. No migration in this batch (Sprint C migration 0014 was the schema add).
- **No frontend changes**. No client-side surface touched.
- **Railway dashboard flip** is operator-controlled. Default OFF means accidental misconfigurations don't accidentally enable IMP-6 — flip is explicit.
- **Production rollback**: instant (unset env var or set 'false' → next process cycle reads False). No data corruption risk; Stage 1 + Stage 2 are additive paths, the legacy single-call path remains intact.
